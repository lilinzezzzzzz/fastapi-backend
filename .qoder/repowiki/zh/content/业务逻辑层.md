## 引言
本文件聚焦于业务逻辑层（Service 层）的实现与设计，围绕用户服务展开，系统性阐述其职责边界、调用关系、接口契约、领域模型、配置项、参数与返回值约定、与其他组件的集成方式，以及异常处理与常见问题的解决方案。特别关注最近的服务层API简化：create_user、get_or_create_user_by_third_party、bind_third_party_account等方法从异步改为同步，这一变化显著简化了服务层的调用方式，提升了开发效率。

## 项目结构
本项目采用分层架构：控制器（Controllers）负责路由与请求入口，业务逻辑（Services）封装领域行为，数据访问（DAO）负责与持久层交互，模型（Models）承载数据结构，工具与基础设施（Infra/Config/Responses/Exceptions/Middlewares）提供支撑能力。

## 核心组件
- 用户服务（UserService）
  - 职责：对外暴露业务方法，协调 DAO 完成用户相关操作；当前包含"hello_world"演示方法与"按手机号查询用户"的业务方法。
  - 依赖注入：通过工厂函数提供单例实例，便于在控制器中注入。
  - 方法概览：
    - hello_world：无参数，返回字符串。
    - get_user_by_phone：接收手机号字符串，返回用户模型或空。
    - create_user：接收用户名、账号、手机号、密码，返回用户模型。
    - get_or_create_user_by_third_party：接收平台名称和第三方用户信息，返回用户模型。
    - bind_third_party_account：接收用户、平台名称和第三方用户信息，无返回值。
  - **更新**：所有方法现已从异步改为同步，移除了 await 关键字的使用，简化了调用方式。
- 用户数据访问（UserDao）
  - 职责：基于 BaseDao 封装查询、计数、更新等通用能力，面向 User 模型提供具体查询方法。
  - 方法概览：
    - get_by_phone：按手机号查询用户。
    - get_by_username：按用户名查询用户。
    - is_phone_exist：判断手机号是否存在。
- 第三方账号数据访问（ThirdPartyAccountDao）
  - 职责：基于 BaseDao 封装第三方账号的查询、计数、更新等通用能力。
  - 方法概览：
    - get_by_platform_and_openid：按平台和 open_id 查询第三方账号。
    - is_platform_openid_exist：检查平台和 open_id 是否存在。
    - get_by_user_id_and_platform：按用户ID和平台查询第三方账号。
    - get_all_by_user_id：获取用户的所有第三方账号。
    - delete_by_user_id_and_platform：删除用户的指定平台账号。
- 用户模型（User）
  - 职责：ORM 映射，定义用户表结构字段。
  - 字段：用户名、账号、手机号、密码哈希。
- 第三方账号模型（ThirdPartyAccount）
  - 职责：ORM 映射，定义第三方账号关联表结构字段。
  - 字段：用户ID、平台名称、open_id、union_id、头像、昵称、令牌信息、额外数据。
- 控制器（API/User/Auth）
  - 职责：定义路由、参数、返回值，注入 Service 并调用其方法，最终通过统一响应体返回。
  - 用户API控制器：提供用户相关的RESTful接口。
  - 认证控制器：处理用户登录、注册、登出等认证相关操作。
- 基础设施与配置
  - 数据库：异步引擎、会话管理、SQL 监听与慢查询日志。
  - 配置：环境变量加载、敏感字段解密、数据库/Redis 连接串生成。
  - 响应：统一响应体、错误码、SSE 辅助。
  - 异常：AppException 自定义异常类，支持任意状态码。
  - 错误码：GlobalErrors 类提供标准化错误代码定义，包含流处理专用错误码。
  - 中间件：认证（Token/签名）、CORS、GZip、请求记录。
  - 流处理：专门的流处理工具函数，提供超时控制和异常处理。

## 组件详解

### 用户服务（UserService）
- 设计要点
  - 通过构造函数注入 UserDao 实例，遵循依赖倒置原则，便于单元测试替换。
  - 业务方法尽量只接受必要参数，避免携带请求对象，提升可测试性与内聚性。
  - 提供工厂函数 new_user_service，集中管理依赖注入与单例。
  - **更新**：所有方法现已改为同步实现，移除了异步修饰符，简化了调用方式。
- 方法说明
  - hello_world
    - 输入：无
    - 返回：字符串
    - 用途：演示服务方法的最小实现
  - get_user_by_phone
    - 输入：手机号（字符串）
    - 返回：用户模型或空
    - 业务规则：按手机号精确查询，若不存在返回空
  - get_user_by_username
    - 输入：用户名（字符串）
    - 返回：用户模型或空
    - 业务规则：按用户名精确查询，若不存在返回空
  - verify_password
    - 输入：用户模型、待验证密码（字符串）
    - 返回：布尔值
    - 业务规则：验证用户密码哈希
  - create_user
    - 输入：用户名、账号、手机号、密码
    - 返回：用户模型
    - 业务规则：创建新用户，包含手机号唯一性检查和密码加密
  - get_or_create_user_by_third_party
    - 输入：平台名称、第三方用户信息
    - 返回：用户模型
    - 业务规则：根据第三方信息获取或创建用户，处理第三方账号绑定
  - bind_third_party_account
    - 输入：用户模型、平台名称、第三方用户信息
    - 返回：无
    - 业务规则：将第三方账号绑定到现有用户，处理重复绑定检查
- 依赖关系
  - 依赖 UserDao 完成用户数据查询
  - 依赖 ThirdPartyAccountDao 完成第三方账号操作
  - 间接依赖 BaseDao 的查询构建器与会话提供者

### 用户模型（User）
- 字段定义
  - username：字符串，最大长度64
  - account：字符串，最大长度64
  - phone：字符串，固定长度11

- 作用

## 统一错误管理系统

### GlobalErrors 类设计
新增的 GlobalErrors 类提供了标准化的错误代码定义，实现了错误码的集中管理和跨模块共享。该类将错误码按照客户端错误（40000-49999）和服务端错误（50000-59999）进行分类管理。

- 客户端错误（40000-49999）
  - BadRequest：40000，请求参数错误
  - Unauthorized：40001，未授权，请登录
  - InvalidSignature：40002，签名验证失败
  - Forbidden：40003，权限不足，禁止访问
  - NotFound：40004，资源不存在
  - PayloadTooLarge：40005，请求载荷过大
  - UnprocessableEntity：40006，无法处理的实体
  - StreamTimeout：40800，流超时（新增）
- 服务端错误（50000-59999）
  - InternalServerError：50000，服务器内部错误
  - StreamError：50001，流处理错误（新增）

### 错误码定义迁移
错误码定义已从 internal/core/exception.py 迁移到 internal/core/errors.py，这一变更带来了以下改进：

- **集中管理**：所有错误码定义集中在单一文件中，便于维护和查找
- **类型安全**：使用 AppError 类型确保错误码的类型安全
- **多语言支持**：每个错误码都包含中英文双语文案
- **命名空间隔离**：errors 实例提供统一的访问接口

### 在业务中的应用
错误码在各个业务模块中得到广泛应用：

- **认证服务**：使用 Unauthorized、BadRequest 等错误码处理认证失败场景
- **中间件**：在认证中间件中使用 InvalidSignature、Unauthorized 等错误码
- **异常记录器**：在全局异常处理中使用标准化错误码
- **流处理**：使用 StreamTimeout 和 StreamError 处理流式响应的超时和错误

## 流处理异常管理

### StreamTimeoutError 异常类
StreamTimeoutError 是专门用于处理流式响应中超时情况的异常类。当流中的单个 chunk 生成超过指定超时时间时抛出。

- 设计特点
  - 继承自 Exception，便于与标准异常处理机制兼容
  - 包含超时时间信息，便于调试和日志记录
  - 支持上下文信息存储，便于传递额外的调试数据
- 构造函数参数
  - message：异常消息描述
  - timeout：超时时间（秒）
  - context：上下文字典，包含调试信息
- 使用场景
  - 非 SSE 模式的流处理，需要业务层自行处理异常
  - 流式 API 调用超时处理
  - 第三方服务响应超时处理

### StreamError 异常类
StreamError 是专门用于处理流式响应中一般性错误的异常类。当流处理过程中发生除超时以外的异常时抛出。

- 设计特点
  - 继承自 Exception，包含原始异常信息
  - 支持原始异常链，便于追踪错误源头
  - 支持上下文信息存储，便于传递调试数据
- 构造函数参数
  - message：异常消息描述
  - original_error：原始异常对象
  - context：上下文字典，包含调试信息
- 使用场景
  - 非 SSE 模式的流处理，需要业务层自行处理异常
  - 流式数据处理过程中的各种异常
  - 第三方服务调用过程中的异常处理

## 流处理工具函数

### stream_with_chunk_control 函数
stream_with_chunk_control 是一个专门用于流式响应的超时控制工具函数，基于 AnyIO 提供单个 chunk 的超时控制。

- 主要功能
  - 对异步生成器的每个 chunk 设置独立的超时时间
  - 支持两种模式：SSE 模式和非 SSE 模式
  - 自动处理超时和异常情况
- 参数说明
  - generator：异步生成器，产生流式数据
  - chunk_timeout：单个 chunk 超时时间（秒）
  - is_sse：是否为 SSE 格式输出
    - True：超时/错误时 yield SSE 错误数据并记录日志
    - False：超时/错误时抛出 StreamTimeoutError/StreamError
- 返回值
  - 异步迭代器，产生处理后的数据项
- 异常处理
  - 超时：抛出 StreamTimeoutError 或返回 SSE 错误数据
  - 异常：抛出 StreamError 或返回 SSE 错误数据
  - 取消：正常传播取消异常
- 使用示例
  - SSE 模式：自动处理错误响应，适合前端直接消费
  - 非 SSE 模式：业务层自行处理异常，适合复杂业务逻辑

## 性能考量
- 数据库连接池
  - 连接池大小、超时与回收策略已在数据库初始化中配置，建议结合压测结果调整。

  - 使用查询构建器的等值/范围/排序组合，避免 N+1 查询；必要时引入子查询或批量查询。
- 响应序列化
  - 统一使用高性能 JSON 序列化，减少序列化开销。
- 中间件
  - GZip 压缩与请求记录有助于网络传输与可观测性，注意在高并发场景下的 CPU 开销平衡。
- 错误码处理
  - 标准化的错误码定义减少了重复代码，提高了错误处理的一致性和性能。
- 流处理性能
  - AnyIO 的超时控制机制提供了高效的异步流处理能力
  - SSE 模式下的错误处理避免了业务层的重复代码
  - 非 SSE 模式下的异常抛出便于业务层进行精细控制
- **更新**：服务层同步方法减少了异步调度开销，提升了简单业务场景的响应速度

## 故障排查指南
- 配置加载失败
  - 症状：启动时报错，提示缺少 .secrets 或环境配置文件
  - 处理：确认 APP_ENV、.secrets 与对应 .env.* 文件存在且格式正确
- 数据库未初始化
  - 症状：获取会话时报错，提示数据库未初始化
  - 处理：确保在 lifespan 中调用初始化函数，或在使用前显式初始化
- 认证失败
  - 症状：内部接口签名失败或外部接口 Token 无效
  - 处理：核对签名头与 Token，确认中间件白名单与路径前缀配置
- 响应异常
  - 症状：统一响应体返回类型不匹配
  - 处理：确保返回数据为 dict/list/Pydantic 模型或 None
- 错误码不一致
  - 症状：不同模块使用不同的错误码定义
  - 处理：统一使用 errors 实例访问标准化错误码，避免重复定义
- 流处理超时
  - 症状：流式响应超时，出现 StreamTimeoutError
  - 处理：检查 chunk_timeout 设置，优化数据生成速度，考虑增加超时时间
- 流处理错误
  - 症状：流式响应异常，出现 StreamError
  - 处理：检查原始异常信息，修复数据生成逻辑，确保异常链完整
- **更新**：服务层同步方法调用
  - 症状：控制器中使用 await 调用同步方法报错
  - 处理：移除 await 关键字，直接调用同步方法

  - 标准化错误码：通过 errors 实例访问 GlobalErrors 定义的错误码
  - AppException：自定义异常类，承载错误码与消息
  - 流处理异常：StreamTimeoutError 和 StreamError 专门处理流式响应
  - 多语言支持：错误码包含中英文双语文案
