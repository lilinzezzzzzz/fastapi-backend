## 引言
本文件聚焦于业务逻辑层（Service 层）的实现与设计，围绕用户服务展开，系统性阐述其职责边界、调用关系、接口契约、领域模型、配置项、参数与返回值约定、与其他组件的集成方式，以及异常处理与常见问题的解决方案。特别关注最新的异常处理增强功能，包括异常链支持、调试效率提升等改进。目标是帮助初学者快速理解整体架构与实现要点，同时为有经验的开发者提供足够的技术深度。

## 项目结构
本项目采用分层架构：控制器（Controllers）负责路由与请求入口，业务逻辑（Services）封装领域行为，数据访问（DAO）负责与持久层交互，模型（Models）承载数据结构，工具与基础设施（Infra/Config/Responses/Exceptions/Middlewares）提供支撑能力。

## 核心组件
- 用户服务（UserService）
  - 职责：对外暴露业务方法，协调 DAO 完成用户相关操作；当前包含"hello_world"演示方法、"按手机号查询用户"和"用户注册"的业务方法。
  - 依赖注入：通过工厂函数提供单例实例，便于在控制器中注入。
  - 方法概览：
    - hello_world：无参数，返回字符串。
    - get_user_by_phone：接收手机号字符串，返回用户模型或空。
    - get_user_by_username：接收用户名字符串，返回用户模型或空。
    - verify_password：验证用户密码，返回布尔值。
    - create_user：创建新用户，包含密码加密与手机号唯一性检查。
- 用户数据访问（UserDao）
  - 职责：基于 BaseDao 封装查询、计数、更新等通用能力，面向 User 模型提供具体查询方法。
  - 方法概览：
    - get_by_phone：按手机号查询用户。
    - get_by_username：按用户名查询用户。
    - is_phone_exist：判断手机号是否存在。
- 用户模型（User）
  - 职责：ORM 映射，定义用户表结构字段。
  - 字段：用户名、账号、手机号、密码哈希等。
- 控制器（Service/Web/Internal/Auth）
  - 职责：定义路由、参数、返回值，注入 Service 并调用其方法，最终通过统一响应体返回。
  - 特别：新增认证控制器，支持用户注册、登录、登出等功能。
- 基础设施与配置
  - 数据库：异步引擎、会话管理、SQL 监听与慢查询日志。
  - 配置：环境变量加载、敏感字段解密、数据库/Redis 连接串生成。
  - 响应：统一响应体、错误码、SSE 辅助。
  - 异常：全局错误码与自定义异常封装，支持异常链。
  - 中间件：认证（Token/签名）、CORS、GZip、请求记录，增强异常日志记录。

## 组件详解

### 用户服务（UserService）
- 设计要点
  - 通过构造函数注入 UserDao 实例，遵循依赖倒置原则，便于单元测试替换。
  - 业务方法尽量只接受必要参数，避免携带请求对象，提升可测试性与内聚性。
  - 提供工厂函数 new_user_service，集中管理依赖注入与单例。
  - 支持密码验证与加密，确保用户信息安全。
- 方法说明
  - hello_world
    - 输入：无
    - 返回：字符串
    - 用途：演示服务方法的最小实现
  - get_user_by_phone
    - 输入：手机号（字符串）
    - 返回：用户模型或空
    - 业务规则：按手机号精确查询，若不存在返回空
  - get_user_by_username
    - 输入：用户名（字符串）
    - 返回：用户模型或空
    - 业务规则：按用户名精确查询，若不存在返回空
  - verify_password
    - 输入：用户模型、原始密码（字符串）
    - 返回：布尔值
    - 业务规则：验证密码哈希匹配
  - create_user
    - 输入：用户名、账号、手机号、原始密码
    - 返回：用户模型
    - 业务规则：检查手机号唯一性，加密密码后创建用户
- 依赖关系
  - 依赖 UserDao 完成数据查询与创建
  - 间接依赖 BaseDao 的查询构建器与会话提供者

## 异常处理增强

### 异常链支持
项目现已全面支持异常链（Exception Chaining），特别是在注册流程中实现了完整的异常上下文保留机制。异常链允许在抛出新异常时保留原始异常的完整堆栈信息，显著提升调试效率。

#### 注册流程异常链实现
在用户注册接口中，异常处理采用了标准的 Python 异常链语法：

```python
try:
    # 创建用户逻辑
    user = await user_service.create_user(...)
    # 登录逻辑
except ValueError as e:
    # 手机号已存在等业务错误
    raise AppException(errors.BadRequest, message=str(e)) from e
except Exception as e:
    # 其他未知错误
    raise AppException(errors.InternalError, message="注册失败，请稍后重试") from e
```

#### 异常链传播机制
异常链通过 `from e` 语法实现，确保：
- 保留原始异常的完整堆栈跟踪
- 提供清晰的异常层次结构
- 支持嵌套异常的完整上下文信息

#### 增强的异常日志记录
中间件层增强了异常日志记录能力：

```python
@staticmethod
def _log_exception(exc: Exception) -> None:
    """
    记录异常日志，根据异常类型使用不同的日志级别
    """
    if isinstance(exc, AppException):
        logger.warning(f"Business exception, exc={get_business_exec_tb(exc)}")
    elif isinstance(exc, RequestValidationError):
        logger.warning(f"Validation Error: {exc}")
    else:
        logger.error(f"Unexpected exception, exc={get_unexpected_exec_tb(exc)}")
```

## 性能考量
- 数据库连接池
  - 连接池大小、超时与回收策略已在数据库初始化中配置，建议结合压测结果调整。
- 查询优化
  - 使用查询构建器的等值/范围/排序组合，避免 N+1 查询；必要时引入子查询或批量查询。
- 响应序列化
  - 统一使用高性能 JSON 序列化，减少序列化开销。
- 中间件
  - GZip 压缩与请求记录有助于网络传输与可观测性，注意在高并发场景下的 CPU 开销平衡。
- 异常处理性能
  - 异常链会增加堆栈信息的处理开销，但在调试收益面前是值得的。
  - 建议在生产环境中合理使用异常链，避免过度捕获和重新抛出。

## 故障排查指南
- 配置加载失败
  - 症状：启动时报错，提示缺少 .secrets 或环境配置文件
  - 处理：确认 APP_ENV、.secrets 与对应 .env.* 文件存在且格式正确
- 数据库未初始化
  - 症状：获取会话时报错，提示数据库未初始化
  - 处理：确保在 lifespan 中调用初始化函数，或在使用前显式初始化
- 认证失败
  - 症状：内部接口签名失败或外部接口 Token 无效
  - 处理：核对签名头与 Token，确认中间件白名单与路径前缀配置
- 响应异常
  - 症状：统一响应体返回类型不匹配
  - 处理：确保返回数据为 dict/list/Pydantic 模型或 None
- 异常链问题
  - 症状：异常信息过于冗长或难以理解
  - 处理：检查异常链的使用是否合理，避免不必要的异常转换
- 调试异常
  - 症状：无法定位异常源头
  - 处理：利用异常链的完整堆栈信息，配合中间件的日志记录进行调试
