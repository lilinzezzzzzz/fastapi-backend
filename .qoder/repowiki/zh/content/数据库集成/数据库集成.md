## 简介
本文件系统化梳理本项目的数据库集成方案，涵盖配置与连接池管理、多数据库支持、数据库抽象层设计（适配器模式）、SQLAlchemy Async ORM 使用与配置、DDL 脚本与初始化流程、事务管理、监控与故障恢复等。**重大更新**：现已扩展支持向量数据库功能，包括向量存储、相似度搜索、嵌入器集成和多后端适配，形成关系数据库与向量数据库的统一集成架构。

**重大更新**：ModelMixin.update方法已从同步重构为异步实现，支持execute参数控制执行模式，提供更灵活的双模式操作能力。

## 核心组件
- 配置与驱动生成：根据 DB_TYPE 动态拼装 SQLAlchemy 异步驱动 URI，支持 mysql、postgresql、oracle。
- **连接管理**：独立的 connection.py 模块提供 init_async_db()、close_async_db()、reset_async_db() 等连接管理功能。
- **会话管理**：get_session() 和 get_read_session() 提供主库和只读副本的会话获取，支持自动降级。
- **连接池配置**：主库 pool_size=10，读库 pool_size=20，支持预连接检查和超时控制。
- **SQL 监控**：事件监听器监控 SQL 执行时间和慢查询日志。
- ORM 抽象层：Base 与 ModelMixin 提供统一模型基类、默认字段、批量插入、软删/恢复、上下文默认值注入。
- **异步更新操作**：ModelMixin.update 支持双模式操作，execute=False仅更新对象属性，execute=True执行SQL更新。
- 查询构建器：QueryBuilder、CountBuilder、UpdateBuilder 提供链式条件、分页、去重计数、软删同步更新。
- DAO 层：BaseDao 封装常用查询、计数、更新入口，提供 execute_transaction 手动事务执行器。
- JSON 类型：JSONType 跨数据库适配，自动选择最优存储（JSONB/JSON/CLOB/TEXT），并提供变更追踪。
- 初始化与生命周期：FastAPI lifespan 中初始化数据库，退出时释放资源；支持慢 SQL 监控与日志。
- **向量数据库后端**：支持 Milvus 等主流向量数据库，提供统一的向量存储和搜索接口。
- **嵌入器模块**：支持 OpenAI 等 LLM 嵌入器，提供文本向量化能力。
- **向量仓储模式**：提供统一的向量数据访问抽象，支持多租户隔离和批量操作。
- **Zvec 向量存储**：嵌入式向量数据库，支持多向量字段和复杂索引类型。

### 连接管理与会话获取
**重大更新**：连接管理功能已重构为独立的 connection.py 模块，提供更清晰的职责分离。

- **初始化管理**：init_async_db() 支持 echo 参数覆盖，幂等初始化，避免重复创建。
- **连接池配置**：主库 pool_pre_ping、pool_size=10、max_overflow=20、pool_timeout=30、pool_recycle=1800。
- **读库连接池**：pool_pre_ping、pool_size=20（更大）、max_overflow=30、pool_timeout=30、pool_recycle=1800。
- **会话工厂**：new_async_session_maker(expire_on_commit=False, autoflush=True)。
- **关闭与重置**：close_async_db() 异步释放，reset_async_db() 同步重置全局变量（Celery 场景）。
- **会话获取**：get_session() 和 get_read_session() 提供上下文管理器，支持自动降级。
- **SQL 监控**：注册 before/after 事件，基于 settings.SLOW_SQL_THRESHOLD 与 DEBUG 输出慢查询日志。

### 读写分离与自动降级

### ORM 抽象层与模型
**重大更新**：ModelMixin.update方法已重构为异步实现，提供双模式操作能力。

- Base 与 ModelMixin：统一声明式基类，提供 id、creator_id、created_at、updater_id、updated_at、deleted_at 等默认字段。
- 工厂方法：create() 自动清洗列、填充默认值；save()/update() 严格区分插入/更新状态。
- **异步更新操作**：update() 支持双模式操作，execute=False仅更新对象属性但不执行SQL，execute=True执行SQL更新。
- **参数验证**：execute=True时要求提供session_provider参数，execute=False时仅更新内存状态。
- 批量操作：insert_rows()（字典）、insert_instances()（实例）高性能批量写入。
- 软删/恢复：soft_delete()/restore() 自动同步 updated_at 与 updater_id。
- 字段反射：has_column()/get_column_names()/has_*_column() 辅助列检测与元数据工具。

### 查询/计数/更新构建器
**重大增强**：查询构建器经过重大重构，引入了更安全的语句访问机制和类型安全保障。

- **增强的类型注解**：所有构建器类都使用泛型类型注解 `T: ModelMixin`，提供更好的 IDE 支持和类型安全性。
- **改进的错误处理**：每个异步方法都包含详细的异常处理和错误消息，便于调试和问题诊断。

## 性能与优化
- 连接池参数建议
  - pool_pre_ping：启用以自动剔除失效连接。
  - **主库 pool_size=10**：根据并发请求数与数据库承载能力设定。
  - **读库 pool_size=20**：读库通常承载更多查询，连接池更大。
  - max_overflow：限制溢出连接数量，避免瞬时洪峰。
  - pool_timeout：设置合理超时，避免阻塞等待。
  - pool_recycle：定期回收连接，降低长时间占用导致的资源泄漏风险。
- JSON 序列化
  - PostgreSQL/Oracle 原生 JSON 模式无需额外序列化，减少 CPU 开销。
  - 其他数据库使用 orjson，注意批量写入时的序列化成本。
- SQL 监控
  - 通过 before/after 事件统计耗时，结合 SLOW_SQL_THRESHOLD 与 DEBUG 控制日志级别。
- 批量写入
  - 优先使用 ModelMixin.insert_rows()/insert_instances()，减少往返次数。
- 分页与索引
  - QueryBuilder.paginate() 与 DDL 中的索引配合，提升查询性能。
- 事务与 flush
  - execute_transaction() 中按需 flush 获取自增 ID，避免不必要的提交延迟。
- **读写分离优化**
  - **读库连接池更大**：读库连接池 size 为 20，主库为 10，适应读多写少场景。
  - **智能降级**：未配置读库时自动回退到主库，确保服务连续性。
  - **分离的会话提供者**：BaseDao 支持独立的读库会话提供者，实现真正的读写分离。
- **异步更新优化**
  - **execute=False模式**：适用于需要先更新内存状态再进行复杂业务逻辑的场景，避免不必要的数据库往返。
  - **execute=True模式**：适用于直接持久化更新的场景，确保数据一致性。
- **向量数据库性能优化**
  - **索引选择**：根据数据规模和查询模式选择合适的索引类型（HNSW 适合大规模近似搜索，IVF 适合超大规模数据）。
  - **批量操作**：使用 upsert_batch 和 embed_texts 批量处理，减少 API 调用次数。
  - **缓存策略**：对频繁查询的向量结果进行缓存，减少重复计算。
  - **维度优化**：根据应用场景选择合适的向量维度，在精度和性能之间找到平衡。

## 故障排查指南
- 初始化失败
  - 确认 DB_TYPE 与驱动映射一致，检查 .env.* 与 .secrets 配置。
  - 核对 sqlalchemy_database_uri 生成逻辑，特别是 Oracle Service Name 与密码转义。
  - **读写分离配置**：检查 DB_READ_HOST 是否正确配置，未配置时会自动降级到主库。
- 连接池问题
  - 观察 pool_timeout 与 max_overflow 设置是否合理；必要时增大或调整。
  - **读库连接池**：读库连接池默认 size=20，如出现连接不足可适当增大。
  - 使用 reset_async_db() 清理 Celery 任务中的事件循环绑定冲突。
- SQL 性能
  - 开启 DEBUG 或调整 SLOW_SQL_THRESHOLD，定位慢查询；结合索引与分页优化。
  - **读写分离场景**：确认查询是否使用了正确的会话提供者，避免读到脏数据。
- 事务异常
  - 使用 execute_transaction() 包裹复杂逻辑，确保异常时自动回滚。
- JSON 类型异常
  - 检查数据库类型与 oracle_native_json 配置；确认容错逻辑生效。
- **异步更新异常**
  - **execute=True但缺少session_provider**：如果调用 update(execute=True) 但未提供 session_provider，会抛出 ValueError。
  - **execute=False模式**：适用于不需要立即持久化的场景，但需要注意内存状态与数据库状态的同步。
  - **状态检查异常**：如果对新创建的对象调用 update()，会抛出 RuntimeError，提示应先调用 insert()。
- **构建器异常**
  - **stmt 属性异常**：如果遇到 "Statement not initialized" 错误，确保在使用 `stmt` 属性前已经初始化了相应的查询条件。
  - **类型不匹配异常**：如果 `select_stmt` 或 `count_stmt` 抛出类型错误，检查是否正确使用了对应的构建器类型。
  - **空列表异常**：`in_()` 方法现在会拒绝空列表输入，需要显式处理空值情况。
- **读写分离异常**
  - **会话提供者为空**：如果 get_read_session() 抛出 "Database is not initialized" 错误，检查数据库是否正确初始化。
  - **写后读一致性**：对于需要强一致性的场景，使用 write_querier 或 write_session_provider。
- **向量数据库异常**
  - **后端连接失败**：检查 VECTOR_BACKEND 配置和后端服务可用性。
  - **嵌入器调用失败**：验证 EMBEDDING_API_KEY 和模型名称配置。
  - **维度不匹配**：确保向量维度与嵌入器配置一致。
  - **索引创建失败**：检查索引参数配置和数据类型支持。

## 结论
本项目的数据库集成采用"配置驱动 + 连接管理 + 抽象层 + 构建器 + DAO"的分层设计，结合 SQLAlchemy Async ORM 实现高性能、可维护的数据访问。通过多数据库驱动映射、连接池参数化、SQL 监控与事务执行器，满足生产级需求。DDL 脚本与 ORM 基类协同，确保结构演进与代码一致性。

**重大更新总结**：
- **连接管理重构**：连接管理功能已迁移到独立的 connection.py 模块，提供更清晰的职责分离。
- **读写分离增强**：支持主库和只读副本的独立连接池，提供智能字段回退和优雅降级机制。
- **动态连接URI生成**：sqlalchemy_read_database_uri 属性根据数据库类型动态生成只读副本连接字符串。
- **异步更新重构**：ModelMixin.update方法已重构为异步实现，支持execute参数控制双模式操作。
- **双模式操作**：execute=False仅更新对象属性但不执行SQL，execute=True执行SQL更新，提供更灵活的操作方式。
- **安全性提升**：新增 `stmt` 属性提供受保护的语句访问，防止未初始化使用。
- **类型安全**：分离的 `select_stmt` 和 `count_stmt` 属性确保类型正确性。
- **功能分离**：排序和分组方法专门用于查询构建，避免类型混淆。
- **软删除改进**：UpdateBuilder 的软删除逻辑更加可靠和一致。
- **类型注解增强**：完整的泛型类型支持提供更好的开发体验。
- **向量数据库集成**：新增完整的向量数据库支持，包括多后端适配、嵌入器集成和统一抽象接口。
- **向量仓储模式**：提供统一的向量数据访问抽象，支持多租户隔离和批量操作。
- **Zvec 向量存储**：嵌入式向量数据库，支持多向量字段和复杂索引类型。

建议在生产环境中进一步完善迁移与版本管理策略，并持续优化连接池与查询性能。向量数据库功能的加入为项目提供了更强大的 AI 应用支持能力。

```python
    # 如果 DB_READ_HOST 未配置，这里会使用主库连接
    users = await session.execute(select(User)).scalars().all()
```

### 读写分离最佳实践
**连接池配置优化**：
- 主库：pool_size=10，适合写操作较多的场景
- 读库：pool_size=20，适合高并发查询场景
- 两个连接池独立配置，避免相互影响

**会话提供者选择**：
- 读操作：优先使用 read_session_provider 或 get_read_session()
- 写操作：始终使用 session_provider 或 get_session()
- 强一致性：写后读场景使用 write_querier 或 write_session_provider

**异步更新最佳实践**：
- **execute=False模式**：适用于需要先更新内存状态再进行复杂业务逻辑的场景
- **execute=True模式**：适用于直接持久化更新的场景，确保数据一致性
- **参数验证**：确保execute=True时提供有效的session_provider
- **状态检查**：对新对象使用insert()，对现有对象使用update()

**向量数据库最佳实践**：
- **索引选择**：根据数据规模选择合适的索引类型，HNSW 适合中小规模近似搜索，IVF 适合大规模数据
- **批量操作**：使用批量 API 减少网络往返，提高写入效率
- **维度管理**：根据应用场景选择合适的向量维度，在精度和性能间平衡
- **缓存策略**：对频繁查询的结果进行缓存，减少重复计算
- **监控告警**：建立向量数据库的监控指标，及时发现性能问题

**监控与诊断**：
- 通过 SQL 监控事件观察读写分离效果
- 使用 SLOW_SQL_THRESHOLD 定位性能瓶颈
- 监控连接池利用率，适时调整 pool_size 参数
- **向量数据库监控**：监控向量搜索性能、索引构建进度、存储空间使用情况
